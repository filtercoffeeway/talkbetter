TalkBetter — Build Specification

A local app that helps a non-native speaker improve spoken English: fewer filler words, better pace, correct grammar, clearer communication, and a more American accent. This document is self-contained: an engineer (or an independent Claude session) should be able to build the app end to end from it.

Version 0.1 · Status: scaffolded, awaiting implementation · Owner: Mahesh

How to use this doc: Build in phase order (1 → 4). Each phase ends with acceptance criteria — do not move on until they pass. Phase 1 runs fully local with no API keys, so you get a working loop before signing up for anything. The repo is already scaffolded with stub modules whose docstrings point back to the relevant phase here.

1. Overview & goals

TalkBetter records the user speaking, then returns structured feedback across four dimensions. The user runs it on their own machine; their audio never has to leave it except for the optional cloud calls in Phases 2–3.

GoalWhat "done" looks likeDifficultyPhase
Reduce filler wordsCounts of "um/uh/like/you know" etc. with a per-minute rateEasy1
Better pace & pausesWords-per-minute + count of long silencesEasy1
Correct grammarCorrected transcript + list of fixes with explanationsMedium2
Clearer communicationClarity score + concrete rephrase suggestionsMedium2
American accentPhoneme-level pronunciation scores; flagged problem wordsHard3

Two interaction modes (both already stubbed in App.jsx):

  • Free speaking — user talks about anything. Drives Phases 1 & 2.
  • Accent practice — user reads a known reference sentence aloud. Required for Phase 3, because Azure pronunciation scoring is reference-based (it compares what was said to expected text).

3. Architecture & data flow

Browser (React) FastAPI backend ───────────────── ──────────────────────────────── Recorder ──▶ audio Blob (webm) │ ▼ POST /api/analyze (multipart: audio[, reference_text]) │ └──────────────────────────────▶ routers/analysis.py │ ┌────────┼─────────────────────────────┐ ▼ ▼ ▼ ▼ transcription filler_pace llm_feedback pronunciation (Whisper) (pure py) (Phase 2) (Phase 3, Azure) │ │ │ │ └────────┴───────┬────────┴─────────────┘ ▼ AnalysisResponse (JSON) ┌───────────────────────────────────────────┘ ▼ FeedbackReport renders sections that are present

The endpoint is additive: Phase 1 returns transcript + pace_fillers; later phases populate language and accent. The frontend renders whatever sections are non-null, so partially-built backends still work.

5. API contract

POST /api/analyze

Request — multipart/form-data:

FieldTypeRequiredNotes
audiofileyeswebm/opus from MediaRecorder (or wav/mp3)
reference_textstringnoPresent only in accent mode; triggers Phase 3 scoring
profile_idintnoPhase 4. If given, the session is saved under that profile and the response includes session_id. Omit for a one-off, non-persisted analysis.

Response — application/json, shape defined authoritatively in backend/app/models/schemas.py. Summary:

{
  "transcript":   { "text": str, "words": [{word,start,end}], "duration_sec": float },
  "pace_fillers": { "words_per_minute": float, "total_words": int, "duration_sec": float,
                    "long_pauses": int, "fillers": [{word,count}],
                    "filler_total": int, "filler_rate_per_min": float },
  "language":     null | { "corrected_text": str,
                           "grammar_issues": [{original,suggestion,explanation}],
                           "clarity": { "score": 0-100, "summary": str, "suggestions": [str] } },
  "accent":       null | { "accuracy_score","fluency_score","completeness_score","pron_score": float,
                           "problem_words": [{word,accuracy,error_type,phonemes:[{phoneme,accuracy}]}] },
  "session_id":   null | int            // Phase 4: set when profile_id was supplied
}

Phase 4 — profiles & progress No API key

Multiple people share one local install via named profiles (no auth — local, trusted). Each analyzed recording is persisted under a profile in a local SQLite DB (data/talkbetter.db, stdlib sqlite3, gitignored). Names are unique case-insensitively.

EndpointPurpose
GET /api/profilesList profiles with session_count.
POST /api/profiles {"name": str}Create (or return existing) profile; 201.
GET /api/history?profile_id=<id>That profile's sessions (newest first) + current_streak (consecutive practice days).
// GET /api/history response
{
  "profile_id": int,
  "current_streak": int,                // consecutive days practiced, incl. today/yesterday
  "sessions": [ { "id","profile_id","created_at","mode","duration_sec","transcript",
                  "words_per_minute","total_words","long_pauses","filler_total",
                  "filler_rate_per_min","clarity_score","pron_score" } ]   // newest first
}

Keep schemas.py and this section in lockstep. The frontend trusts these field names.

8. Phase 1 — Transcription + pace + fillers No API key

Outcome: Record speech → see a transcript, words-per-minute, long-pause count, and filler-word breakdown. Fully local. This is the first end-to-end loop; get it solid before anything else.

8.1 Transcription — services/transcription.py

  • Lazy-load a single WhisperModel(settings.whisper_model, device=..., compute_type=...) at module import (model load is expensive; do it once).
  • Browser audio is webm/opus. Write the uploaded bytes to a temp file; faster-whisper decodes via its bundled av/ffmpeg. (Optionally transcode to 16kHz mono WAV first for reliability.)
  • Call model.transcribe(path, word_timestamps=True). Iterate segments, then segment.words → build WordTiming(word, start, end) list.
  • Return Transcript(text=joined, words=[...], duration_sec=info.duration).

8.2 Pace & fillers — services/filler_pace.py

  • words_per_minute = total_words / (duration_sec/60).
  • Fillers: case-insensitive count of single-word fillers AND multi-word phrases ("you know", "i mean", "sort of", "kind of") via a sliding window over the token stream.
  • long_pauses: walk consecutive words; count gaps where next.start - prev.end > PAUSE_THRESHOLD_SEC (default 1.5s).
  • filler_rate_per_min = filler_total / (duration_sec/60) so clips of different lengths are comparable.
Acceptance criteria —
  • Recording a clip in the UI returns a populated transcript + pace_fillers.
  • Un-skip and pass backend/tests/test_filler_pace.py (pytest).
  • GET /api/health returns {"status":"ok"}; /docs shows the endpoint.
  • No API keys required for any of the above.

9. Phase 2 — Grammar + clarity LLM key

Outcome: The transcript is corrected, mistakes are listed with short explanations, and clarity is scored with actionable rephrase tips.

Implementation — services/llm_feedback.py

  • Dispatch on settings.llm_provider; keep provider calls behind one function so swapping is a config change.
  • System prompt: frame the model as an ESL speaking coach for a non-native speaker targeting natural, clear American English. Ask it to be encouraging and specific.
  • Require strict JSON matching LanguageReport (corrected_text, grammar_issues[], clarity{score,summary,suggestions[]}). Parse defensively; on failure, retry once asking for JSON only.
  • Wire it into routers/analysis.py (uncomment the Phase 2 line).
Acceptance criteria —
  • A clip with deliberate grammar errors returns a sensible corrected_text and ≥1 grammar issue.
  • clarity.score is 0–100 with a non-empty summary.
  • Switching LLM_PROVIDER between anthropic/openai works without code changes beyond the key.
  • Malformed model output never 500s the endpoint (graceful fallback).

10. Phase 3 — American accent Azure key

Outcome: In accent-practice mode, the user reads a reference sentence and gets overall + per-word + per-phoneme pronunciation scores, with problem words surfaced.

Implementation — services/pronunciation.py

  • Create an Azure Speech resource; put key + region in .env.
  • Transcode the uploaded webm to 16kHz mono PCM WAV with ffmpeg (Azure requirement).
  • Build PronunciationAssessmentConfig(reference_text, GradingSystem.HundredMark, Granularity.Phoneme, enable_miscue=True) and run recognition.
  • From NBest[0].PronunciationAssessment pull AccuracyScore / FluencyScore / CompletenessScore / PronScore, plus per-word and per-phoneme scores.
  • Flag words with accuracy below a threshold (e.g. <60) as problem_words.
  • Frontend already has accent mode + reference textarea; replace the placeholder with a real sentence bank (categorize by the sounds non-native speakers find hard: th, r/l, v/w, vowel length).
Acceptance criteria —
  • Reading the reference sentence returns scores in 0–100 and a problem-word list.
  • Deliberately mispronouncing a word lowers its score and lists it as a problem word.
  • Free-speaking mode (no reference_text) still works and simply omits the accent section.

11. Phase 4 — Progress tracking & polish Optional

Outcome: The app becomes a practice habit, not a one-shot analyzer.

  • Persist each session (timestamp, scores, filler rate) to a local SQLite DB; add GET /api/history.
  • Frontend dashboard: charts of filler-rate, WPM, clarity, and pron-score over time (Chart.js). Show streaks.
  • Daily practice prompts / sentence-of-the-day; optionally a scheduled reminder.
  • Export a session report (the docx/pdf skills can generate a shareable summary).

No new external dependencies required for the core of this phase.

12. Future / out of scope (v1)

  • Mobile app — wrap the React frontend with Capacitor/React Native once the web version is proven. Native mic + offline Whisper are the main considerations.
  • Real-time feedback — streaming transcription / live filler alerts (websockets).
  • Conversation practice — an LLM role-play partner that also scores you.
  • Multi-user / auth — only needed if this stops being a personal local tool.

End of specification. Build in order, honor the acceptance criteria, and keep schemas.py as the single source of truth for the API shape.
